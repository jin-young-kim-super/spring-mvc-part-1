MVC의 한계(여기서의 MVC는 서블릿과 JSP만을 이용해 만든 MVC 패턴을 말한다)
결론부터 말하면, 컨트롤러의 중복 부분(공통 처리 부분)이 많다라는 것이다. 
1. 포워드 중복
RequestDispatcher dispatcher = request.getRequestDispatcher(viewPath);
dispatcher.forward(request, response);
-> 물론 이 부분을 메서드화해서 공통 처리해도 되지만, 일일이 컨트롤러 내에서 메서드를 호출해야 하며, 무엇보다 개발자의 실수로 【메서드 호출 누락】이 발생할 수가 있다. 
또한 중요한 것은 이렇게 똑같은 메서드를 각 컨트롤러에서 호출하는 것 자체가 중복 부분이다. 

2. 사용되지 않는 response 문제

3. ViewPath에 중복 
String viewPath = "/WEB-INF/views/new-form.jsp";
prefix: /WEB-INF/views/
suffix: .jsp
-> 만약에 jsp파일에서 thymeleaf 파일로 변경해야 한다면 모든 jsp를 사용하는 컨트롤러에서 수정을 해줘야 한다. 

즉, 공통 처리가 어렵다는 것이다. 
-> 이러한 MVC의 한계로 인해 탄생한 것이 Front Controller라는 각 컨트롤러의 공통처리 부분을 담당하는 컨트롤러를 만들어서 
요청 발생 시, 각 컨트롤러에 들어가기 전에 , 공통 처리 부분을 Front Controller를 거쳐서 설정하여 그 다음에 각 컨트롤러에서 로직을 실행하게 하면 된다. 
※공통 처리 예시 : 로그 기능 등

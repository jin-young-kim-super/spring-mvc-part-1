우리가 어떤 기능을 분리할까 말까할 때 하나의 기준이 되는 것이 있다 .
바로, 라이프사이클의 차이이다. 
※라이프 사이클 : 시간에 따라, 누가, 왜, 언제 바뀌는지!
즉 다음 요소가 모두 포함된다. 
・Who changes it (누가 바꾸는가)
・Why it changes (왜 바꾸는가)
・When / Frequency (언제 / 얼마나 자주)
・Impact scope (변경 영향 범위)

비지니스 로직의 라이프 사이클
누가 : 백엔드 개발자
왜(or 무엇)    : 서비스 정책/검증
언제   : 서비스 정책/검증 변경 시


HTML의 라이프 사이클
누가 : 프론트 개발자
왜(or 무엇) : UI
언제         : UI 변경 시

예를 들어, jsp 파일 안에 비지니스 로직과 html이 하나의 파일에 있다고 하자. (즉, 하나의 파일에 2개의 라이프 사이클이 섞여 있는 경우)
그럼 아래의 유지/보수 문제가 생긴다. 
1. 불필요한 사이드 이펙트 발생
<% if(user.isAdmin()) { %>
    <div>Admin Page</div>
<% } %>
누가 : 프론트엔드 개발자 
왜(무엇) : Admin Page를 User Page로 변경
-> 이 과정에서 if 문등을 실수로 수정하여 비지니스 로직에 영향을 줄 수가 있다. 
이 원인은 비니지스 로직은 백엔드 개발자(누가) 서비스 정책/검증의 다른 라이프사이클인데 같이 섞여 있기 때문이다.
변경 영향 범위가 전혀 관련 없는 비지니스 로직에도 영향을 줄 가능성이 있다. 

2. 협업이 어려워 진다. 
HTML과 비지니스 로직이 섞여 있으면, 프론트 개발자는 비지니스 로직도 이해해야 하며, 백엔드 개발자는 html을 이해하고 있었야 한다. 
즉, 모든 걸 알고 있어야 하므로 분업이 어렵고 개발 생산성이 줄어든다. 
-> 하나의 파일에 서로 다른 라이프사이클이 섞여 있으면 프론트/백엔드 개발자(누가)는 서로 다른 라이프사이클의 코드를 전부 이해하고 있어야 한다. 

3. 재사용성 불가능
하나의 파일에 자바 코드와 html이 있으므로 예를 들어 http API로써 재사용이 불가능하다. 
특히 모바일 앱으로 확장해서 사용 불가능
-> 서로 목적이 다른 2개의 라이프사이클이 서로 강하게 결합이 되어 있기 때문이다. 
주로, 똑같은 비지니스 로직을 서비스 여러군데에서 재사용된다. 

4. 테스트 코드 작성 불가능
테스트를 하려면 서버를 띄워서 일일이 인간이 눈으로 봐야 한다. 
테스트 자동화 불가능. 

5. 가독성 문제
스파게티 코드가 되기 쉬우므로 가독성이 0이다. 


항상 변경 이유를 기준으로 분리를 해라(SRP원칙)
-> 분리할까 말까 고민될 떄에는 분리 대상이 다른 변경 이유에 의해서 개발자의 실수 등의 이유로 변경될 가능성이 있는지를 생각해라. 
가능성이 있다면 분리하며 다른 변경 이유에 의해서 분리 대상의 코드가 개발자의 실수 등의 이유로 변경될 가능성을 차단하다.

jsp vs servlet
순수 서블릿으로 개발할 떄에는 자바 객체안에 HTML 코드가 들어 가 있었다.
그런데 html 한줄 한줄 자바 코드를 이용해서 추가해줘야 했다. 이러한 불편한 때문에
기본적으로는 HTML 파일을 기준으로 하고,그 HTML 중에서 필요한 부분만 자바 코드를 이용해서 바꿀 수 있다면 좀더 편하지 않을까 해서 나온 것이
JSP 템플릿 엔진이였다. 실제로 jsp 파일을 봐라!!! 동적인 HTML 부분에만 자바 코드가 들어 가 있다.
그리고 JSP 파일에서 착각할수 있는 것이 우리가 예를 들어, /jsp/members/save.jsp에 접속했을 때, "어? 난 서블릿 객체에서 urlpattern 설정을 안 했는데
어떻게 save.jsp파일을 찾아가지라고 생각할 수 있는데, 위에서도 언급을 하였지만 jsp파일은 결국에는 서블릿 객체로 변환이 되기 때문에 본질적으로는
/jsp/members/save.jsp로 url 매핑된 서블릿 객체가 있는 것이다. 정적 HTML을 찾는 것이 아니다."
만약 jsp가 없었다면 save.jsp파일에서 각 html문을 println() 등의 자바 코드를 이용해서 지저분하게 코딩을 해야 했다.
이게 HTML 가독성에 있어서 순수 서블릿 개발보다 훨씬 좋다. 왜냐하면 동적인 부분에만 자바 코드를 넣었기 때문.
그리고 jsp 파일을 보면 알겠지만, jsp 파일은 HTML 출력 전용 파일이라는 인사이트도 가지고 있자.
또한 순수 서블릿 객체 내에서 html을 생성하면 html 태그의 개행등을 개발자가 일일이 println 같은 거를 써줘야 되면 들여 쓰기 같은 것도 일일이 자바 코드로 계산해야 한다.
그래서 브라우저 등에서 html을 열었을 때 코드가 매우매우 지저분하지만 jsp는 브라우저에서 열었을 떄 그런 일이 적다라는 인사이트도 가지고 있자.


서블릿과 jsp의 한계
서블릿의 불편한 점을 해결하기 위해서 jsp가 등장했다.
jsp는 서블릿의 한계를 잘 극복하고 있다. 그러나 save.jsp 파일을 보아라. 윗부분에는 비지니스 로직이 있고, 아래에는 html 코드가 있다.
지금이야 코드가 짧아서 괜찮지만 유지/보수를 거치면서 코드량이 방대해 지게 되면 가독성독 떨어 지고, html 수정 중 윗부분의 자바 코드도 건들여야 하는데 이때 코드수가
많아지면 실수하기 너무 쉽다. 그리고 무엇보다도 하나의 파일을 여러 사람이 수정하게 되면 commit시에 충돌도 많이 일어 날 것이다. 예를 들어, 윗부분에서 자바 코드를 수정하다가
필요에 의해서 html를 수정했다고 하자. 근데 html 수정 작업을 담당하는 사람이 그 부분을 수정했다고 해보자. 이러면 충돌이 일어나서 유지/보수가 개헬이 된다.

위와 같은 한계점을 극복하기 위해서 나온 것이 바로 MVC 패턴이다.
Controller
Model : Controller에서 View로 데이터 전달
VIEW
-> 즉, HTML파일과 비지니스 로직을 각각 VIEW, Controller로 분리를 한 것이다. 그럼 의문이 들 것이다. HTML을 자바 코드로 동적으로 변경할 떄 필요한 데이터는 어떻게 하죠?
그걸 해결한 것이 바로 Model이다. Model은 비지니스 로직 실행 완료 후 VIEW에서 필요한 데이터를 Model 객체에 담아서 VIEW에서 사용할 수 있게 해주는 것이다.
그로 인해서 JSP의 원래 목적은 HTML 동적 변경에만 집중할 수 있게 된 것이다. 물론 지금은 JSP 템플릿을 사용하지 않고 타임리프 등의 다른 템플릿 엔진을 사용한다.
